# Multi-User + Drizzle ORM Migration Design

Date: 2026-06-09

## 1. Multi-User System

### 1.1 User Model

- **roles**: `admin` and `user`.
- The initial admin is created by the existing password-setup flow (first login → set password → becomes admin).
- Admins create/delete users and manage their own accounts. Users cannot create accounts.
- Each user has their own credentials (username + password, Argon2id).
- Session tokens include the username and role.

### 1.2 Data Isolation (Granularity A)

- **Resources per user**: each account (X/GitHub/GitLab/Reddit) is owned by exactly one dashboard user.
- Normal users see **only their own connected platform accounts**.
- Admins see **all platform accounts across all users**, but:
  - `auth_token` is never returned in API responses for other users' accounts.
  - The API returns `owner_username` alongside `screen_name` for admin visibility.

### 1.3 Schema Changes

New `users` table:

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,   -- Argon2id
  role TEXT NOT NULL DEFAULT 'user',  -- 'admin' | 'user'
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Add `owner_id INTEGER NOT NULL REFERENCES users(id)` to `accounts`.

Single-user mode (no `users` table): all existing accounts get `owner_id = 1` (the bootstrap admin). Password hash moves from `config.json` into `users.password_hash`.

### 1.4 Bootstrap

On **first run**:
1. No users exist → single-user mode.
2. First password set via Settings UI → creates admin user (id=1), sets password_hash on that row.
3. All existing accounts (from migration) are already assigned `owner_id = 1`.

### 1.5 API Changes

- `POST /api/auth/login` accepts `{ username, password }`. Default admin username is `admin`.
- `GET /api/auth/me` returns `{ authenticated, username, role }`.
- `POST /api/users` (admin only) creates a user.
- `DELETE /api/users/:id` (admin only) deletes a user and their platform accounts.
- `GET /api/users` (admin only) lists users (no password hashes).
- All data queries filter by `owner_id` for normal users, return all for admins.
- Every destructive action (delete account, delete user) requires a confirmation token:
  - Client calls `POST /api/confirm-token` → server returns `{ token: "a3b2c1" }` (6 random alphanumeric chars).
  - Client shows the token, user types it into a text field.
  - Client sends the destructive request with `{ confirmToken: "a3b2c1" }`.
  - Server validates, consumes the token (one-time use, 5-minute expiry), then performs the action.

## 2. Drizzle ORM Migration

### 2.1 Goal

Replace raw SQL in `server/db.ts` with Drizzle ORM for schema definition and queries. Keep SQLite as the only active backend for now; PostgreSQL support is wired in config but not yet tested.

### 2.2 Approach

- **Use `drizzle-orm` + `drizzle-kit`** for schema definitions, migrations, and type-safe queries.
- **Keep existing data**: Drizzle's `push` or migration system can run against the existing SQLite file without data loss. We'll write manual migration SQL where needed.
- **Schema files** move to `db/schema/` — one file per domain (users, accounts, twitter, github, gitlab, reddit).
- **Database driver selection** via `config.json`:
  ```json
  {
    "database": {
      "driver": "sqlite",         // "sqlite" | "postgresql"
      "sqlite": { "path": "data/db/dashboard.db" },
      "postgresql": { "host": "...", "port": 5432, "database": "...", "user": "...", "password": "..." }
    }
  }
  ```
- On startup, read `config.database.driver` → instantiate the correct Drizzle client.
- Drizzle migrations live in `db/migrations/` (auto-generated by `drizzle-kit`).

### 2.3 Query Layer

- `server/db.ts` is refactored into `db/queries/` — one file per domain, all using Drizzle's typed query builders.
- All existing query functions keep the same signatures — callers (`routes/*.ts`, `fetchers/*.ts`) are updated only for multi-user filtering (owner_id), not for Drizzle API changes.

### 2.4 Migration Safety

1. Backup `dashboard.db` to `dashboard.db.backup-YYYYMMDD-HHMMSS` before running migrations.
2. Write a **one-shot migration script** that Drizzle's migrate can run: adds `users` table, adds `owner_id` column with default 1, creates the admin user from `config.json` password hash.
3. If migration fails, restore from backup.
4. After successful migration, remove `passwordHash` from `config.json` (it's now in `users.password_hash`).

## 3. Confirmation Dialog Component

A reusable `ConfirmDialog` component in `client/src/components/ui/`:

- Props: `open`, `onOpenChange`, `title`, `description`, `onConfirm`, `confirmLabel` (default "删除").
- On open, calls `POST /api/confirm-token` to get a random 6-char token.
- Displays the token prominently (monospace, highlighted background).
- User types the token into an input field.
- Confirm button is disabled until the typed text matches the token exactly.
- On confirm, calls `onConfirm(confirmToken)` — caller passes it to the destructive API.

## 4. Implementation Order

1. **Install Drizzle** — `drizzle-orm`, `drizzle-kit`, `@libsql/client` (for SQLite).
2. **Define Drizzle schemas** in `db/schema/` — all existing tables + new `users` table.
3. **Database connection** in `server/db.ts` — Drizzle client, driver selection from config.
4. **Migration script** — one-shot: create users table, add owner_id, migrate password.
5. **Refactor queries** to Drizzle — `db/queries/` per domain, keep function signatures.
6. **Multi-user auth** — update login/logout/me, add user CRUD routes.
7. **Multi-user data isolation** — add owner_id filtering to all data queries.
8. **ConfirmDialog** — reusable component, wire to delete account and delete user.
9. **Admin UI** — user management page (list, create, delete users).
10. **Clean up** — remove old raw SQL, remove passwordHash from config.json schema.
